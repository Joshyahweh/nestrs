The nestrs command-line tool scaffolds projects and generates source files so you spend less time writing boilerplate. The crates.io package is published as **`nestrs-scaffold`** (the name `nestrs-cli` is taken), but the binary you run is **`nestrs`**.

## Install

```bash
cargo install nestrs-scaffold
nestrs --help
```

<Note>
If you are working inside a clone of the nestrs repository, the workspace defines a Cargo alias so you can run the CLI without a global install: `cargo nestrs doctor`, `cargo nestrs generate resource items --transport rest --path src`, and so on.
</Note>

## Commands overview

| Command | Purpose |
|---------|---------|
| `nestrs new <name>` | Create a new single-crate app with `Cargo.toml`, `src/main.rs`, a starter module and controller, `.env.example`, and an optional git init. |
| `nestrs generate <kind> <name>` | Generate a controller, service, module, DTO, guard, pipe, filter, interceptor, strategy, resolver, gateway, microservice, transport, or a full resource scaffold. |
| `nestrs doctor` | Print `rustc` and `cargo` versions, scan `Cargo.toml` for nestrs feature hints, and check `src/**/*.rs` for common misconfigurations. |

## `nestrs new`

`nestrs new` creates a project directory, writes an opinionated `Cargo.toml` and `src/main.rs`, generates a `.env.example`, a `Dockerfile`, a `.gitignore`, and initializes a git repository.

```bash
nestrs new my-api
cd my-api
cargo run
```

The generated `src/main.rs` includes a full `NestFactory` builder chain with a global prefix, request IDs, request tracing, Prometheus metrics, and a health check endpoint — ready to extend.

### Flags

| Flag | Description |
|------|-------------|
| `--no-git` | Skip `git init` — useful when you manage git separately or use a monorepo. |
| `--strict` | Add `#![deny(unsafe_code)]` to the generated `src/main.rs`. |
| `--package-manager cargo` | Explicit package manager selection. Only `cargo` is supported; omit this flag for the default. |

<Info>
`nestrs new` only supports `--package-manager cargo`. There is no npm/yarn/pnpm integration — nestrs stays within the standard Cargo toolchain.
</Info>

### Generated project layout

```
my-api/
├── Cargo.toml        # nestrs + tokio + serde dependencies, release profile
├── src/
│   └── main.rs       # AppModule, AppController, NestFactory builder chain
├── .env              # PORT, NESTRS_ENV, RUST_LOG, DATABASE_URL defaults
├── .env.example      # Committed env template
├── .gitignore        # /target and .env
└── Dockerfile        # Multi-stage build with health check
```

## `nestrs generate`

Use `nestrs generate` (or the short alias `nestrs g`) to create individual source files. Run with `--dry-run` first when you are unsure about the output path.

### Generator kinds

| Kind | Short alias | What it generates |
|------|-------------|-------------------|
| `resource` | `res` | Module + service + controller/resolver/gateway + two DTOs + `mod.rs` |
| `service` | `s` | `#[injectable]` service struct with a stub `impl` |
| `controller` | `co` | `#[controller(prefix = "...")]` struct |
| `module` | `mo` | `#[module(...)]` struct with controller and provider slots |
| `dto` | `dto` | `#[dto]` struct with `#[IsString]` and `#[Length]` example fields |
| `guard` | `gu` | `CanActivate` trait implementation |
| `pipe` | `pi` | `PipeTransform<String>` trait implementation |
| `filter` | `fi` | `ExceptionFilter` trait implementation |
| `interceptor` | `in` | `Interceptor` trait implementation |
| `strategy` | `st` | Strategy struct with a `name()` method |
| `resolver` | `r` | GraphQL resolver struct (stub) |
| `gateway` | `ga` | WebSocket gateway struct (stub) |
| `microservice` | `ms` | Microservice struct with a `transport()` method |
| `transport` | `tr` | Transport struct (stub) |

### Common flags

All generators accept the following flags:

| Flag | Description |
|------|-------------|
| `--style nest\|rust` | File naming convention. `nest` produces `item.controller.rs`; `rust` produces `item_controller.rs`. Default: `nest`. |
| `--path <dir>` | Output directory. Default: `src`. |
| `--dry-run` | Print the files that would be created without writing anything. |
| `--force` | Overwrite existing files. Without this flag, the generator errors if a file already exists. |
| `--quiet` | Suppress all output. |

### `nestrs generate resource`

`resource` is the most powerful generator — it creates a full feature directory with a module, service, entry point (controller, resolver, gateway, or transport depending on the transport flag), two DTOs, and a `mod.rs`.

```bash
# Dry run first to preview the output tree
nestrs generate resource invoices --transport rest --path src --dry-run

# Write the files
nestrs generate resource invoices --transport rest --path src
```

The generator also appends `pub mod invoices;` and the module use declaration to `src/main.rs`, `src/lib.rs`, or `src/mod.rs` if one of those files exists.

#### `--transport` flag

| Value | Entry point generated |
|-------|-----------------------|
| `rest` | `controller` with `#[routes]` impl (default) |
| `graphql` | `resolver` |
| `ws` | `gateway` |
| `grpc` | gRPC-style transport |
| `microservice` | Message-based transport |

When you run `nestrs generate resource` without `--transport` in an interactive terminal, the CLI prompts you to select a transport. In CI or scripts, pass `--transport` explicitly or add `--no-interactive` to default to REST.

<Tip>
Always run with `--dry-run` the first time you target an unfamiliar directory. The generator rewrites parent module files automatically; preview the changes before committing them.
</Tip>

### Examples

```bash
# Generate a service
nestrs g service users

# Generate a guard
nestrs g guard auth --path src/common

# Generate a REST resource (full scaffold)
nestrs generate resource orders --transport rest --path src

# Generate a GraphQL resource
nestrs generate resource posts --transport graphql --path src

# Dry run a microservice resource
nestrs g res notifications --transport microservice --dry-run
```

## `nestrs doctor`

`nestrs doctor` checks your local toolchain and current project for common issues:

- Prints `rustc` and `cargo` versions
- Scans `Cargo.toml` for nestrs feature hints
- Heuristically checks `src/**/*.rs` for misconfigurations — for example, calling `enable_openapi()` without the `openapi` feature enabled in `Cargo.toml`

```bash
nestrs doctor
```

<Warning>
`nestrs doctor` is a hint tool. It does not replace `cargo check`, `cargo clippy`, or your CI matrix. Use its output as a starting point for diagnosing build or feature issues.
</Warning>

## Comparison with Nest CLI

nestrs intentionally stays closer to Cargo and normal Rust workflows rather than duplicating the features Nest layers on Node.

| Nest CLI area | nestrs support | What to use instead |
|---------------|----------------|---------------------|
| `nest new` / app skeleton | Partial — `nestrs new` creates a single-crate starter | `nestrs new` covers the common case |
| `nest generate` (CRUD, modules) | Partial — overlapping generators; naming and file trees differ | `nestrs generate` |
| Workspaces (multiple apps/libs) | Not supported | [Cargo workspaces](https://doc.rust-lang.org/book/ch14-03-cargo-workspaces.html): one root `Cargo.toml` with `[workspace].members` |
| Libraries (`@nestjs/...`-style packages) | Not supported | `cargo new --lib`, workspace members, crates.io |
| Scripts (`npm run ...`) | Not supported | `cargo run --bin <name>`, [cargo-make](https://github.com/sagiegurari/cargo-make), [just](https://github.com/casey/just), or shell scripts |
| `nest build` / `nest start` | Not supported | `cargo build`, `cargo run`; production: binary + process manager or container |
| Plugins extending the CLI | Not supported | Fork or wrap `nestrs-scaffold`, or generate with your own templates |

This keeps one Rust toolchain story — Cargo — instead of duplicating workspace and script features that Nest layers on top of Node.
