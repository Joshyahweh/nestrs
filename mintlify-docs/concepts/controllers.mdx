---
title: "Controllers: Defining HTTP Route Handlers in nestrs"
description: "Declare controllers with #[controller], define route handlers with #[routes], and use HTTP method macros, extractors, and response helpers in nestrs."
---

Controllers handle incoming HTTP requests and return responses. In nestrs, a controller is a plain Rust struct annotated with `#[controller(prefix = "...")]`. Its route handlers live in an `impl` block annotated with `#[routes(state = T)]`, where `T` is the provider type the handlers need from the DI container. This design keeps routing logic close to its business-logic dependency without any global state.

## Declaring a controller

Annotate a struct with `#[controller(prefix = "...")]` to set the path prefix for all routes in that controller.

```rust
use nestrs::prelude::*;

#[controller(prefix = "/users")]
pub struct UsersController;
```

Register the controller in a module's `controllers` list:

```rust
#[module(
    controllers = [UsersController],
    providers = [UsersService],
)]
pub struct UsersModule;
```

## The `#[routes]` impl block

Route handlers are `async fn` methods inside an `impl` block annotated with `#[routes(state = T)]`. The `state` argument names the provider type that the handlers inject via Axum's `State` extractor.

```rust
#[controller(prefix = "/users")]
pub struct UsersController;

#[routes(state = UsersService)]
impl UsersController {
    #[get("/")]
    async fn list(State(svc): State<Arc<UsersService>>) -> Json<Vec<User>> {
        Json(svc.find_all().await)
    }

    #[get("/:id")]
    async fn find_one(
        Path(id): Path<u64>,
        State(svc): State<Arc<UsersService>>,
    ) -> Json<User> {
        Json(svc.find_one(id).await)
    }

    #[post("/")]
    async fn create(
        State(svc): State<Arc<UsersService>>,
        ValidatedBody(dto): ValidatedBody<CreateUserDto>,
    ) -> Json<User> {
        Json(svc.create(dto).await)
    }
}
```

## HTTP method macros

nestrs re-exports the following route macros from `nestrs-macros`. Place them on handler methods inside a `#[routes]` impl block.

| Macro | HTTP method | Example |
|-------|-------------|---------|
| `#[get("path")]` | GET | `#[get("/")]` |
| `#[post("path")]` | POST | `#[post("/")]` |
| `#[put("path")]` | PUT | `#[put("/:id")]` |
| `#[patch("path")]` | PATCH | `#[patch("/:id")]` |
| `#[delete("path")]` | DELETE | `#[delete("/:id")]` |
| `#[options("path")]` | OPTIONS | `#[options("/")]` |
| `#[head("path")]` | HEAD | `#[head("/")]` |
| `#[all("path")]` | All methods | `#[all("/probe")]` |

The path argument is appended to the controller prefix. A controller with `prefix = "/users"` and a handler `#[get("/:id")]` matches `GET /users/:id`.

## Extractors

nestrs uses Axum's extractor model. List any number of extractors as handler parameters — Axum resolves them in declaration order.

<Tabs>
  <Tab title="Route parameters">
    Use Axum's `Path` extractor to capture named segments from the URL.

    ```rust
    #[get("/:id")]
    async fn find_one(Path(id): Path<u64>) -> Json<User> {
        // id is the captured :id segment parsed to u64
        Json(User { id, ..Default::default() })
    }
    ```

    For multiple segments, use a tuple or a struct:

    ```rust
    #[get("/:org/:repo")]
    async fn get_repo(Path((org, repo)): Path<(String, String)>) -> Json<Repo> {
        Json(Repo { org, repo })
    }
    ```
  </Tab>
  <Tab title="Query parameters">
    Use Axum's `Query` extractor (or `ValidatedQuery` for automatic validation) to read query string values.

    ```rust
    #[derive(serde::Deserialize)]
    struct ListQuery {
        page: Option<u32>,
        limit: Option<u32>,
    }

    #[get("/")]
    async fn list(Query(q): Query<ListQuery>) -> Json<Vec<User>> {
        let page = q.page.unwrap_or(1);
        let limit = q.limit.unwrap_or(20);
        Json(vec![]) // replace with real query
    }
    ```
  </Tab>
  <Tab title="Request body">
    Use `ValidatedBody` to deserialize and validate a JSON body against a DTO that implements `serde::Deserialize` and `validator::Validate`.

    ```rust
    #[derive(serde::Deserialize, validator::Validate)]
    pub struct CreateUserDto {
        #[validate(length(min = 1))]
        pub name: String,
        #[validate(email)]
        pub email: String,
    }

    #[post("/")]
    async fn create(
        State(svc): State<Arc<UsersService>>,
        ValidatedBody(dto): ValidatedBody<CreateUserDto>,
    ) -> Json<User> {
        Json(svc.create(dto).await)
    }
    ```

    For unvalidated JSON, use Axum's `Json<T>` extractor directly.
  </Tab>
  <Tab title="State injection">
    Use Axum's `State` extractor to receive the provider registered in `#[routes(state = T)]`.

    ```rust
    #[get("/")]
    async fn list(State(svc): State<Arc<UsersService>>) -> Json<Vec<User>> {
        Json(svc.find_all().await)
    }
    ```

    The provider is always wrapped in `Arc<T>` because nestrs registers singletons as `Arc`.
  </Tab>
</Tabs>

## Response types

Handler return types must implement `axum::response::IntoResponse`. The most common options are:

<CardGroup cols={2}>
  <Card title="Json<T>" icon="brackets-curly">
    Serializes `T` as JSON with `Content-Type: application/json`. Requires `T: serde::Serialize`.
  </Card>
  <Card title="&'static str / String" icon="text">
    Returns a plain text body with `Content-Type: text/plain`.
  </Card>
  <Card title="StatusCode" icon="hash">
    Returns an empty response with a specific status code.
  </Card>
  <Card title="(StatusCode, Json<T>)" icon="layer-group">
    Returns a JSON body with a custom status code using a tuple.
  </Card>
</CardGroup>

## Custom status codes

Use `#[http_code(N)]` on a handler to set a non-200 default status code for successful responses.

```rust
#[post("/")]
#[http_code(201)]
async fn create(
    State(svc): State<Arc<UsersService>>,
    ValidatedBody(dto): ValidatedBody<CreateUserDto>,
) -> Json<User> {
    Json(svc.create(dto).await)
}
```

## Custom response headers

Use `#[response_header("Header-Name", "value")]` to attach a static header to every response from that handler.

```rust
#[get("/export")]
#[response_header("Content-Disposition", "attachment; filename=\"users.csv\"")]
async fn export(State(svc): State<Arc<UsersService>>) -> String {
    svc.export_csv().await
}
```

## Redirects

Use `#[redirect("/new/path")]` to return a redirect response. The default status is 302 unless combined with `#[http_code(301)]`.

```rust
#[get("/old-path")]
#[redirect("/new-path")]
async fn old_route() {}
```

## Versioned routes

You can assign a version to an individual handler with `#[ver("v2")]`, or to the entire controller with `#[controller(prefix = "/users", version = "v1")]`. Use `#[version("v2")]` as a separate attribute when you need to declare the version before `#[controller]`:

```rust
#[version("v2")]
#[controller(prefix = "/users")]
pub struct UsersV2Controller;
```

<Note>
URI-based versioning requires calling `.enable_uri_versioning("v1")` on your `NestApplication`. Header and media-type versioning are enabled with `.enable_header_versioning(...)` or `.enable_media_type_versioning(...)` respectively.
</Note>
