---

Providers are the services, repositories, and helpers that your controllers and other providers depend on. nestrs resolves provider dependencies automatically through a compile-time and runtime DI container called the `ProviderRegistry`. When you annotate a struct with `#[injectable]`, nestrs generates an `Injectable` impl that wires up the struct's `Arc<T>` fields from the registry — no manual constructors needed.

## The `#[injectable]` macro

Annotate a struct with `#[injectable]` to make it available for injection. Fields must be typed as `Arc<T>`, where `T` is another provider registered in the same module or an imported module.

```rust
use nestrs::prelude::*;

#[injectable]
pub struct UsersService {
    db: Arc<DatabaseService>,
}
```

`#[injectable]` generates:

```rust
impl Injectable for UsersService {
    fn construct(registry: &ProviderRegistry) -> Arc<Self> {
        Arc::new(UsersService {
            db: registry.get::<DatabaseService>(),
        })
    }
}
```

Register the provider in a module and it becomes available to any controller or provider in that module (and in importing modules if exported):

```rust
#[module(
    providers = [UsersService, DatabaseService],
    exports = [UsersService],
)]
pub struct UsersModule;
```

## Injection scopes

`ProviderScope` controls how long a resolved instance lives. Set the scope with the `scope` argument on `#[injectable]`.

| Scope | Attribute | Lifetime |
|-------|-----------|----------|
| Singleton (default) | `#[injectable]` | One instance per application container |
| Transient | `#[injectable(scope = "transient")]` | New instance per injection site |
| Request | `#[injectable(scope = "request")]` | One instance per HTTP request |

### Singleton (default)

The default scope. `construct` is called once; the same `Arc<T>` is reused across all injection sites and for the lifetime of the application.

```rust
#[injectable]
pub struct CacheService {
    client: Arc<RedisClient>,
}
```

### Transient

A new instance is created every time the provider is injected. Use transient scope for stateless utilities or when each caller needs independent state.

```rust
#[injectable(scope = "transient")]
pub struct RequestLogger;
```

<Note>
Lifecycle hooks (`on_module_init`, etc.) only run for **singleton** providers. Transient providers are constructed on demand and do not receive the global hook sequence.
</Note>

### Request scope

One instance is created per HTTP request and shared within that request's task-local context. Use request scope for providers that carry per-request state such as the current user or a request-specific database transaction.

<Steps>
  <Step title="Enable request scope on the application">
    Call `use_request_scope()` on your `NestApplication` before listening. This installs the task-local cache middleware that nestrs needs to isolate instances per request.

    ```rust
    NestFactory::create::<AppModule>()
        .use_request_scope()
        .listen_graceful(3000)
        .await;
    ```
  </Step>
  <Step title="Mark the provider with request scope">
    ```rust
    #[injectable(scope = "request")]
    pub struct CurrentUser {
        pub id: u64,
        pub email: String,
    }
    ```
  </Step>
  <Step title="Inject with RequestScoped<T> in handlers">
    Use the `RequestScoped<T>` extractor in your route handlers to receive the per-request instance.

    ```rust
    #[get("/me")]
    async fn get_me(
        RequestScoped(user): RequestScoped<CurrentUser>,
    ) -> Json<UserProfile> {
        Json(UserProfile { id: user.id, email: user.email.clone() })
    }
    ```
  </Step>
</Steps>

<Warning>
Resolving a request-scoped provider outside of a request context (for example from `on_module_init`) will panic. Always enable `use_request_scope()` on the app before using `RequestScoped<T>` in handlers.
</Warning>

## Custom providers

When the automatic `#[injectable]` wiring is not enough, use the `ProviderRegistry` methods directly in a `DynamicModuleBuilder` or `DynamicModule`.

<Tabs>
  <Tab title="useValue">
    Register a pre-built instance as a singleton. No lifecycle hooks run for `useValue` providers.

    ```rust
    use nestrs::core::{ProviderRegistry, DynamicModule};
    use std::sync::Arc;

    fn build_module() -> DynamicModule {
        let mut registry = ProviderRegistry::new();
        registry.register_use_value::<AppConfig>(Arc::new(AppConfig {
            debug: true,
        }));
        DynamicModule::from_parts(registry, axum::Router::new(), vec![])
    }
    ```
  </Tab>
  <Tab title="useFactory">
    Register a provider from a synchronous closure. The closure receives the `ProviderRegistry` so it can resolve its own dependencies.

    ```rust
    use nestrs::core::{ProviderRegistry, ProviderScope};

    registry.register_use_factory::<HeavyService>(
        ProviderScope::Singleton,
        |reg| {
            let light = reg.get::<LightService>();
            Arc::new(HeavyService::new(light))
        },
    );
    ```

    Keep the closure non-async. Defer I/O to `on_module_init` on the service type.
  </Tab>
  <Tab title="useClass">
    `register_use_class` is equivalent to `register` for a normal `#[injectable]` type. It is available when you are building a registry by hand.

    ```rust
    registry.register_use_class::<UsersService>();
    ```
  </Tab>
</Tabs>

## Manual resolution with `ModuleRef`

`ModuleRef` is a handle to the root `ProviderRegistry` after the application is built. Use it for dynamic resolution in plugin systems or conditional code paths where static injection is awkward.

```rust
use nestrs::core::{ModuleRef};

// Obtain ModuleRef before calling into_router or listen:
let app = NestFactory::create::<AppModule>();
let mref: ModuleRef = app.module_ref();
let svc: Arc<UsersService> = mref.get::<UsersService>();
```

<Note>
`ModuleRef::get` panics if the requested type is not registered. If you are unsure whether a provider exists, use `DiscoveryService::get_providers()` to inspect the registry before resolving.
</Note>

## `DiscoveryService`

`DiscoveryService` gives you introspection over the registered providers and HTTP routes. It is useful for diagnostics, admin tooling, or building dynamic OpenAPI documentation.

```rust
use nestrs::core::{DiscoveryService, ModuleRef};

let mref: ModuleRef = app.module_ref();
let discovery = DiscoveryService::new(mref);

// All registered TypeId keys
let provider_ids = discovery.get_providers();

// Human-readable type names (debug/logging)
let type_names = discovery.get_provider_type_names();

// All registered HTTP routes (path, method, handler name)
let routes = discovery.get_routes();
```

## Lifecycle hooks

Implement the `Injectable` trait directly on your struct to override the default no-op lifecycle hooks. All hooks are `async`.

```rust
use nestrs::prelude::*;

#[injectable]
pub struct DatabaseService {
    // pool: Arc<Pool<Postgres>>  // example
}

#[async_trait]
impl Injectable for DatabaseService {
    fn construct(registry: &ProviderRegistry) -> Arc<Self> {
        Arc::new(DatabaseService {})
    }

    async fn on_module_init(&self) {
        // run migrations, connect pool
    }

    async fn on_application_bootstrap(&self) {
        // warm caches, check connectivity
    }

    async fn on_application_shutdown(&self) {
        // flush pending work
    }

    async fn on_module_destroy(&self) {
        // close pool
    }
}
```

The hooks run in this order:

1. `on_module_init` — after all singletons are constructed
2. `on_application_bootstrap` — before the server starts accepting requests
3. `on_application_shutdown` — when a shutdown signal is received
4. `on_module_destroy` — after shutdown completes
