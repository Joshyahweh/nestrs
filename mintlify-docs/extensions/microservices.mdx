---

The `nestrs-microservices` crate turns any module into a message-driven service. You define handlers the same way you define HTTP controllers — with injectable structs and proc-macro decorators — and then choose a transport adapter at boot time. All adapters share one JSON wire format, so you can swap transports without rewriting your business logic.

## Choosing a transport

| Transport | Feature flag | Best for |
|-----------|-------------|----------|
| TCP | `microservices` | Internal services, simple RPC |
| NATS | `microservices-nats` | Low-ops, pub/sub, single region |
| Redis | `microservices-redis` | Existing Redis infrastructure |
| Kafka | `microservices-kafka` | High-throughput event streaming |
| MQTT | `microservices-mqtt` | IoT, mobile fan-out |
| RabbitMQ | `microservices-rabbitmq` | Complex routing, enterprise patterns |
| gRPC | `microservices-grpc` | Cross-language RPC, strong contracts |

## Define handlers with `#[micro_routes]`

Use `#[micro_routes]` on an `impl` block the same way you use `#[routes]` for HTTP. Decorate individual methods with `#[message_pattern]` for request/reply flows or `#[event_pattern]` for fire-and-forget events.

```rust src/orders/orders_handler.rs
use nestrs::prelude::*;

#[derive(serde::Deserialize)]
struct GetOrderRequest {
    id: u64,
}

#[derive(serde::Serialize)]
struct OrderResponse {
    id: u64,
    status: String,
}

#[injectable]
struct OrdersHandler;

#[micro_routes]
impl OrdersHandler {
    #[message_pattern("orders.get")]
    async fn get_order(&self, req: GetOrderRequest) -> OrderResponse {
        OrderResponse { id: req.id, status: "pending".into() }
    }

    #[event_pattern("orders.created")]
    async fn on_order_created(&self, payload: serde_json::Value) {
        tracing::info!("order created: {payload}");
    }
}
```

<Note>
  The method decorated with `#[message_pattern]` must return a serializable value. Methods decorated with `#[event_pattern]` are fire-and-forget and may return `()`.
</Note>

## Cross-cutting concerns on micro handlers

You can attach interceptors, guards, and pipes to micro handlers the same way you do for HTTP — but note the **execution order differs**:

```rust
#[micro_routes]
impl OrdersHandler {
    #[message_pattern("orders.get")]
    #[use_micro_interceptors(LoggingInterceptor)]
    #[use_micro_guards(AuthGuard)]
    #[use_micro_pipes(ValidationPipe)]
    async fn get_order(&self, req: GetOrderRequest) -> OrderResponse {
        // ...
    }
}
```

<Warning>
  On `#[micro_routes]` handlers, execution order is **interceptors → guards → pipes → handler**. This differs from the HTTP pipeline, where guards run before interceptors. Do not assume the two pipelines reorder cross-cutting types the same way.
</Warning>

There is no microservice analogue of HTTP exception filters. Handlers return `Result<_, TransportError>`, and `HttpException` is mapped into `TransportError` in generated code.

## Boot the microservice

Call the transport-specific factory method on `NestFactory`. Each method returns a `MicroserviceApplication` you can optionally configure before calling `.listen()`.

<Steps>
  <Step title="Wire the module">
    ```rust src/main.rs
    use nestrs::prelude::*;

    #[module(providers = [OrdersHandler])]
    struct AppModule;
    ```
  </Step>
  <Step title="Choose a factory method and listen">
    <Tabs>
      <Tab title="TCP">

## Hybrid HTTP + microservice apps

Call `.also_listen_http(port)` on the `MicroserviceApplication` to run an HTTP router and a microservice transport in the same process. Use `.configure_http(...)` to apply HTTP-specific configuration before listening.

```rust src/main.rs
use nestrs::microservices::TcpMicroserviceOptions;
use nestrs::prelude::*;

#[tokio::main]
async fn main() {
    NestFactory::create_microservice::<AppModule>(
        TcpMicroserviceOptions { port: 3001 },
    )
    .also_listen_http(3000)
    .configure_http(|app| app.enable_cors(CorsOptions::permissive()))
    .listen()
    .await;
}
```

## Calling other services with `ClientsModule` and `ClientProxy`

Register remote services with `ClientsModule::register`, then inject `ClientsService` or `ClientProxy` into your providers.

```rust src/orders/orders_module.rs
use nestrs::prelude::*;
use nestrs::microservices::{ClientConfig, ClientsModule, NatsTransportOptions};

#[module(
    imports = [
        ClientsModule::register(&[
            ClientConfig {
                name: "USER_SERVICE",
                transport: NatsTransportOptions::new("nats://127.0.0.1:4222").into(),
            },
        ])
    ],
    providers = [OrdersService],
)]
struct OrdersModule;
```

When exactly one client is registered, `ClientProxy` is also exported as the default client. Look up a named client via `clients.expect("USER_SERVICE")`.

```rust src/orders/orders_service.rs
use nestrs::prelude::*;
use nestrs::microservices::{ClientProxy, ClientsService};
use std::sync::Arc;

#[injectable]
struct OrdersService {
    clients: Arc<ClientsService>,
}

impl OrdersService {
    async fn notify_user(&self, user_id: u64) -> Result<(), nestrs::microservices::TransportError> {
        let proxy: Arc<ClientProxy> = self.clients.expect("USER_SERVICE");

        #[derive(serde::Serialize)]
        struct Payload { user_id: u64 }
        #[derive(serde::Deserialize)]
        struct Ack { ok: bool }

        let _ack: Ack = proxy.send("users.notify", &Payload { user_id }).await?;
        Ok(())
    }
}
```

<Tip>
  Use `proxy.send(pattern, payload)` for request/reply flows where you need a typed response. Use `proxy.emit(pattern, payload)` for fire-and-forget integration events where no reply is required.
</Tip>

## In-process event bus

`EventBus` provides an in-process async event bus with subscribe and emit semantics, re-exported from `nestrs_events`. Use `#[event_routes]` with `#[on_event]` to register handlers that are auto-subscribed when the app boots.

```rust src/notifications/notifications_handler.rs
use nestrs::prelude::*;

#[injectable]
struct NotificationsHandler;

#[event_routes]
impl NotificationsHandler {
    #[on_event("user.created")]
    async fn on_user_created(&self, payload: serde_json::Value) {
        tracing::info!("new user event: {payload}");
    }
}
```

## Reliability guidance

- All transports use **at-least-once delivery** semantics. Write handlers to be idempotent.
- Include a correlation or request ID in event payloads for tracing.
- For critical integration events, use the outbox pattern: write your business state and an outbox row in a single database transaction, then publish outbox rows asynchronously with retries.
- Configure a dead-letter or retry strategy per transport adapter.

## Wire format

All adapters (Redis, Kafka, MQTT, RabbitMQ, custom, and the JSON inside gRPC) share one `WireRequest` / `WireResponse` JSON shape from `nestrs_microservices::wire`. The stability marker is `nestrs_microservices::WIRE_FORMAT_DOC_REVISION`. Run `cargo test -p nestrs-microservices --test wire_conformance` when changing serialization on those types.
