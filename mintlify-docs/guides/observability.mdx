---

nestrs bundles the observability stack you need for a production HTTP service: a Prometheus scrape endpoint, structured `tracing`-based request logs and spans, optional OpenTelemetry span export via OTLP, request ID propagation, and HTTP health probes. Each feature is an opt-in builder method on `NestApplication`, so you compose exactly the pipeline your deployment needs without enabling anything you do not want.

## Tracing subscriber

Install the global `tracing` subscriber by calling `configure_tracing` on your `NestApplication`. Do this once, before `listen`, so all log lines and spans share one pipeline. The subscriber reads its filter directive from `NESTRS_LOG` if set, otherwise `RUST_LOG`, and falls back to `TracingConfig::default_directive` (default `"info"`).

```rust
use nestrs::prelude::*;

#[module]
struct AppModule;

#[tokio::main]
async fn main() {
    let tracing = TracingConfig::builder()
        .format(TracingFormat::Json)           // Pretty for local dev, Json for production
        .default_directive("info,nestrs=debug");

    NestFactory::create::<AppModule>()
        .configure_tracing(tracing)
        .listen(3000)
        .await;
}
```

`TracingFormat::Pretty` produces human-readable multi-line output. `TracingFormat::Json` emits one JSON object per log line, which is the standard format for log aggregation pipelines (Loki, CloudWatch, Datadog, etc.).

<Tip>
Use `TracingFormat::Pretty` during local development and `TracingFormat::Json` in production. You can select the format at runtime by reading an environment variable and branching on it before calling `configure_tracing`.
</Tip>

## Request IDs

`use_request_id` installs `tower-http`'s `SetRequestIdLayer` and `PropagateRequestIdLayer` using the `x-request-id` header. Incoming requests that already carry `x-request-id` are left unchanged; requests without it receive a fresh UUID. The response echoes the same `x-request-id` value.

```rust
use nestrs::prelude::*;

#[tokio::main]
async fn main() {
    NestFactory::create::<AppModule>()
        .configure_tracing(TracingConfig::builder())
        .use_request_id()
        .listen(3000)
        .await;
}
```

## Request tracing middleware

`use_request_tracing` records a structured completion log line and creates a `tracing` span for each request. The span name is `http.server.request` with fields `http.request.method` and `http.route`, aligned with OpenTelemetry HTTP server semantics. The completion line includes `method`, `path`, `status`, `duration_ms`, and `request_id` (when present).

Pass a `RequestTracingOptions` value to skip noisy paths — typically your metrics scrape endpoint and health probes:

```rust
use nestrs::prelude::*;

#[tokio::main]
async fn main() {
    NestFactory::create::<AppModule>()
        .configure_tracing(TracingConfig::builder())
        .use_request_id()
        .use_request_tracing(
            RequestTracingOptions::builder()
                .skip_paths(["/metrics", "/health", "/ready"]),
        )
        .listen(3000)
        .await;
}
```

<Note>
`http.route` in span fields is the literal request URI path, not the Axum route template (e.g., `/users/42` rather than `/users/:id`). This is a known characteristic of the middleware layer's position in the stack.
</Note>

## Prometheus metrics

`enable_metrics` registers a Prometheus scrape endpoint and installs request-level metrics middleware. The built-in metrics are:

| Metric | Type | Labels |
|---|---|---|
| `http_request_duration_seconds` | Histogram | `method` |
| `http_requests_total` | Counter | `method`, `status` |
| `http_requests_in_flight` | Gauge | — |

The histogram uses buckets `[0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1.0, 2.5, 5.0, 10.0]` seconds.

```rust
use nestrs::prelude::*;

#[tokio::main]
async fn main() {
    NestFactory::create::<AppModule>()
        .configure_tracing(TracingConfig::builder())
        .use_request_id()
        .use_request_tracing(
            RequestTracingOptions::builder().skip_paths(["/metrics", "/health"]),
        )
        .enable_metrics("/metrics")
        .enable_health_check("/health")
        .listen(3000)
        .await;
}
```

The `/metrics` path is mounted at the server root — outside `set_global_prefix` and `enable_uri_versioning` — so your Prometheus scrape config does not need to know about your API prefix.

## OpenTelemetry span export

Enable the `otel` Cargo feature and replace `configure_tracing` with `configure_tracing_opentelemetry` to export spans to an OTLP collector (Jaeger, Tempo, Honeycomb, etc.).

```toml
[dependencies]
nestrs = { version = "*", features = ["otel"] }
```

```rust
use nestrs::prelude::*;

#[module]
struct AppModule;

#[tokio::main]
async fn main() {
    let tracing = TracingConfig::builder().format(TracingFormat::Json);
    let otel = OpenTelemetryConfig::new("my-service")
        .endpoint("http://localhost:4317")  // gRPC OTLP endpoint
        .sample_ratio(1.0);                 // sample every request

    NestFactory::create::<AppModule>()
        .configure_tracing_opentelemetry(tracing, otel)
        .use_request_id()
        .use_request_tracing(
            RequestTracingOptions::builder().skip_paths(["/metrics"]),
        )
        .enable_metrics("/metrics")
        .listen(3000)
        .await;
}
```

`OpenTelemetryConfig` builder methods:

| Method | Description |
|---|---|
| `new(service_name)` | Required. Sets `service.name` in the OTLP resource. |
| `.endpoint(url)` | Override the OTLP gRPC endpoint (default: `OTEL_EXPORTER_OTLP_ENDPOINT` or `http://localhost:4317`). |
| `.sample_ratio(f)` | Sampling ratio in `[0.0, 1.0]`. Default `1.0` (always sample). |
| `.timeout(duration)` | Export timeout for the OTLP exporter. |
| `.resource_attr(key, value)` | Add a resource attribute (e.g., `"deployment.environment"`, `"prod"`). |

<Note>
When `configure_tracing_opentelemetry` is used, logs emitted via `tracing::info!` and friends inside an active span automatically carry the same trace context as OTLP spans, because `tracing-opentelemetry` bridges the two pipelines.
</Note>

## Health checks

nestrs provides two HTTP probe routes, both mounted at the server root independent of any global prefix or URI versioning.

<Tabs>
  <Tab title="Liveness probe">
    `enable_health_check` registers a `GET` route that returns `200 {"status":"ok"}`. Use it as a Kubernetes liveness probe to confirm the process is alive.

    ```rust
    use nestrs::prelude::*;

    fn main() {
        let _router = NestFactory::create::<AppModule>()
            .enable_health_check("/health")
            .into_router();
    }
    ```
  </Tab>
  <Tab title="Readiness probe">
    `enable_readiness_check` runs a list of `HealthIndicator` implementations on each request. Returns `200` when all indicators report `HealthStatus::Up` and `503` when any report `HealthStatus::Down`.

    ```rust
    use nestrs::prelude::*;

    #[derive(Clone, Default)]
    struct DbIndicator;

    #[async_trait]
    impl HealthIndicator for DbIndicator {
        fn name(&self) -> &'static str {
            "db"
        }

        async fn check(&self) -> HealthStatus {
            // ping your database here; return Down with a message on failure
            HealthStatus::Up
        }
    }

    #[module]
    struct AppModule;

    fn main() {
        let indicators: Vec<std::sync::Arc<dyn HealthIndicator>> =
            vec![std::sync::Arc::new(DbIndicator)];

        let _router = NestFactory::create::<AppModule>()
            .enable_readiness_check("/ready", indicators)
            .into_router();
    }
    ```

    The response body follows a Terminus-style shape:

    ```json
    {
      "status": "ok",
      "info": { "db": { "status": "up" } },
      "error": {},
      "details": {}
    }
    ```

    When any indicator is down, `status` becomes `"error"` and the `error` object describes the failing checks.
  </Tab>
</Tabs>

## Environment variables

| Variable | Role |
|---|---|
| `NESTRS_LOG` | Tracing filter directive (overrides `RUST_LOG` and `TracingConfig::default_directive`). |
| `RUST_LOG` | Fallback filter when `NESTRS_LOG` is not set. Standard `tracing-subscriber` semantics. |
| `NESTRS_ENV` / `APP_ENV` / `RUST_ENV` | Set to `production` or `prod` to activate `runtime_is_production()`. Used by `enable_production_errors_from_env` and CORS permissive-mode warnings. |
| `OTEL_EXPORTER_OTLP_ENDPOINT` | OTLP collector address when `OpenTelemetryConfig::endpoint` is not set explicitly. |

## Troubleshooting

| Symptom | What to check |
|---|---|
| No log lines appear | `configure_tracing` must be called before `listen`; verify `NESTRS_LOG` / `RUST_LOG` is set and nothing else installed a conflicting global subscriber first. |
| `/metrics` floods request logs | Add `"/metrics"` to `RequestTracingOptions::skip_paths`. |
| Spans missing in Jaeger / Tempo | Confirm the `otel` feature is enabled, the OTLP endpoint URL is correct, and the collector is receiving gRPC traffic on port 4317 (or your configured port). |
| High-cardinality `http.route` labels | Expected: `http.route` is the literal URI path at this middleware layer. Add a custom layer or application metric if you need route-template granularity. |
