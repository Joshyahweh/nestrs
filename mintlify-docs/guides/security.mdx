description: "Add CORS allowlists, security response headers, rate limiting, bearer token guards, CSRF protection, and production error sanitization to nestrs apps."
---

nestrs is explicit over magical: most security controls are opt-in so minimal services stay simple, but every control you need for a production-grade browser-facing API is available on `NestApplication`. This page walks through each layer in the order you would typically add them when hardening an app — CORS, security headers, rate limiting, authentication guards, CSRF protection, and finally production error sanitization.

## CORS

Call `enable_cors` with a `CorsOptions` value to add the `tower-http` CORS layer. By default no CORS headers are added.

<Tabs>
  <Tab title="Development (permissive)">
    `CorsOptions::permissive()` allows any origin, method, and header. Use this only in local development — nestrs emits a `tracing` WARN if permissive CORS is active when `NESTRS_ENV`, `APP_ENV`, or `RUST_ENV` is set to `production` or `prod`.

    ```rust
    use nestrs::prelude::*;

    #[tokio::main]
    async fn main() {
        NestFactory::create::<AppModule>()
            .enable_cors(CorsOptions::permissive())
            .listen(3000)
            .await;
    }
    ```
  </Tab>
  <Tab title="Production (explicit allowlist)">
    Use `CorsOptions::builder()` to specify exact origins, methods, and headers. For APIs that send cookies alongside credentials, also call `.allow_credentials(true)` — but note that browsers reject `*` origins when credentials are enabled, so you must use named origins.

    ```rust
    use nestrs::prelude::*;

    fn main() {
        let _router = NestFactory::create::<AppModule>()
            .enable_cors(
                CorsOptions::builder()
                    .allow_origins(["https://app.example.com"])
                    .allow_methods(["GET", "POST", "PUT", "DELETE"])
                    .allow_headers(["content-type", "authorization"])
                    .allow_credentials(true)
                    .max_age_secs(3600)
                    .build(),
            )
            .into_router();
    }
    ```
  </Tab>
</Tabs>

## Security headers

`use_security_headers` installs response headers via `tower-http`'s `SetResponseHeaderLayer`. Headers are set only when the response does not already include them.

<Tabs>
  <Tab title="Default">
    `SecurityHeaders::default()` sets:

    - `X-Content-Type-Options: nosniff`
    - `X-Frame-Options: DENY`
    - `Referrer-Policy: strict-origin-when-cross-origin`
    - `X-XSS-Protection: 0` (disables the legacy XSS auditor)
    - `Permissions-Policy: geolocation=(), microphone=(), camera=()`

    ```rust
    use nestrs::prelude::*;

    fn main() {
        let _router = NestFactory::create::<AppModule>()
            .use_security_headers(SecurityHeaders::default())
            .into_router();
    }
    ```
  </Tab>
  <Tab title="Helmet-like">
    `SecurityHeaders::helmet_like()` adds everything in `default()` plus:

    - `Cross-Origin-Opener-Policy: same-origin`
    - `Cross-Origin-Resource-Policy: same-origin`
    - `X-DNS-Prefetch-Control: off`
    - `X-Download-Options: noopen`
    - `X-Permitted-Cross-Domain-Policies: none`

    CSP and HSTS are **not** set automatically — configure them explicitly for your deployment.

    ```rust
    use nestrs::prelude::*;

    fn main() {
        let _router = NestFactory::create::<AppModule>()
            .use_security_headers(SecurityHeaders::helmet_like())
            .into_router();
    }
    ```
  </Tab>
  <Tab title="Custom CSP and HSTS">
    Chain builder methods on `SecurityHeaders::helmet_like()` (or `default()`) to set CSP and HSTS.

    ```rust
    use nestrs::prelude::*;

    fn main() {
        let headers = SecurityHeaders::helmet_like()
            .content_security_policy("default-src 'self'; img-src 'self' data:")
            .hsts("max-age=31536000; includeSubDomains");

        let _router = NestFactory::create::<AppModule>()
            .use_security_headers(headers)
            .into_router();
    }
    ```
  </Tab>
</Tabs>

## Rate limiting

`use_rate_limit` adds a fixed-window rate limiter. The default limit is 100 requests per 60-second window, applied across all requests. Clients that exceed the limit receive `429 Too Many Requests`.

<Tabs>
  <Tab title="In-memory (single instance)">
    ```rust
    use nestrs::prelude::*;

    fn main() {
        let _router = NestFactory::create::<AppModule>()
            .use_rate_limit(
                RateLimitOptions::builder()
                    .max_requests(100)
                    .window_secs(60)
                    .build(),
            )
            .into_router();
    }
    ```
  </Tab>
  <Tab title="Redis-backed (distributed)">
    The `cache-redis` feature enables distributed rate limiting using Redis `INCR` + `EXPIRE` keyed by client IP. This shares state across multiple app instances.

    ```rust
    use nestrs::prelude::*;

    fn main() {
        let _router = NestFactory::create::<AppModule>()
            .use_rate_limit(
                RateLimitOptions::builder()
                    .max_requests(200)
                    .window_secs(60)
                    .redis("redis://localhost:6379", "ratelimit")
                    .build(),
            )
            .into_router();
    }
    ```
  </Tab>
</Tabs>

## Guards and roles

nestrs uses `CanActivate` guards for authentication and authorization. Guards run before handler extractors.

### BearerToken extractor

`BearerToken` is an Axum extractor that parses `Authorization: Bearer <token>` and returns `401 Unauthorized` if the header is absent or malformed. `OptionalBearerToken` wraps the result in `Option<String>` and never rejects.

```rust
use nestrs::prelude::*;

#[get("/me")]
async fn get_me(BearerToken(token): BearerToken) -> Json<serde_json::Value> {
    // token is the raw bearer string; validate it with your JWT/auth crate
    Json(serde_json::json!({ "token_prefix": &token[..8] }))
}

#[get("/public-or-authed")]
async fn maybe_authed(
    OptionalBearerToken(token): OptionalBearerToken,
) -> Json<serde_json::Value> {
    Json(serde_json::json!({ "authed": token.is_some() }))
}
```

### AuthStrategyGuard

`AuthStrategyGuard<S>` runs `S::validate` where `S` implements `AuthStrategy` and `Default`. Use this to wire a stateless JWT or API-key validator you own.

```rust
use nestrs::prelude::*;

#[derive(Default)]
struct JwtStrategy;

#[async_trait]
impl AuthStrategy for JwtStrategy {
    async fn validate(
        &self,
        parts: &axum::http::request::Parts,
    ) -> Result<(), AuthError> {
        // parse and verify the JWT from the Authorization header
        let _raw = parts
            .headers
            .get("authorization")
            .ok_or_else(|| AuthError::new("missing token"))?;
        Ok(())
    }
}

#[controller(prefix = "/secure")]
struct SecureController;

#[routes]
#[use_guards(AuthStrategyGuard<JwtStrategy>)]
impl SecureController {
    #[get("/data")]
    async fn data() -> Json<serde_json::Value> {
        Json(serde_json::json!({ "secret": true }))
    }
}
```

### Role-based access with `#[roles]`

Apply `#[roles("admin", "editor")]` to a route and pair it with `XRoleMetadataGuard` to restrict access by role. The guard reads the `x-role` request header and checks it against the roles declared in metadata.

```rust
use nestrs::prelude::*;

#[controller(prefix = "/admin")]
struct AdminController;

#[routes]
#[use_guards(XRoleMetadataGuard)]
impl AdminController {
    #[get("/dashboard")]
    #[roles("admin")]
    async fn dashboard() -> &'static str {
        "admin only"
    }
}
```

<Note>
`XRoleMetadataGuard` reads roles from the `x-role` header — it is a simple demonstration pattern. In production you would derive roles from a validated JWT claim instead of a plain header. Replace it with your own `CanActivate` implementation that calls `route_roles_csv(parts)` to read the declared roles.
</Note>

## CSRF protection

CSRF protection requires the `csrf` Cargo feature (which implies `cookies`). The middleware uses a double-submit pattern: it compares a `csrf_token` cookie value against the `X-CSRF-Token` request header using a constant-time equality check on every POST, PUT, PATCH, and DELETE request.

<Steps>
  <Step title="Enable the feature">
    Add `csrf` to your `nestrs` dependency features in `Cargo.toml`:

    ```toml
    [dependencies]
    nestrs = { version = "*", features = ["csrf"] }
    ```
  </Step>
  <Step title="Wire cookies and CSRF on the app">
    Call `use_cookies()` first, then `use_csrf_protection`. The cookie manager layer must be outermost so the `Cookies` extractor is populated before the CSRF check runs.

    ```rust
    use nestrs::prelude::*;

    fn main() {
        let _router = NestFactory::create::<AppModule>()
            .use_cookies()
            .use_csrf_protection(CsrfProtectionConfig::default())
            .into_router();
    }
    ```
  </Step>
  <Step title="Issue the token on a safe request">
    On a `GET` (or any safe method), set the `csrf_token` cookie and return the value to the client. The client must send the same value back in the `X-CSRF-Token` header on mutations.

    ```rust
    use nestrs::prelude::*;
    use tower_cookies::Cookies;

    #[get("/csrf-token")]
    async fn csrf_token(cookies: Cookies) -> Json<serde_json::Value> {
        let token = uuid::Uuid::new_v4().to_string();
        cookies.add(tower_cookies::Cookie::new("csrf_token", token.clone()));
        Json(serde_json::json!({ "csrfToken": token }))
    }
    ```
  </Step>
</Steps>

<Warning>
Always pair `use_cookies()` with `use_csrf_protection`. If you call `use_cookies()` without configuring CSRF, nestrs emits a `tracing` WARN at router build time. Cookie-authenticated browser clients remain vulnerable to cross-site request forgery on unsafe HTTP methods until CSRF is wired.
</Warning>

## Production error sanitization

`enable_production_errors` strips internal details from `5xx` JSON responses: the `message` field is replaced with `"An unexpected error occurred"` and the `errors` field is removed. Use `enable_production_errors_from_env` to activate this automatically when `NESTRS_ENV`, `APP_ENV`, or `RUST_ENV` equals `production` or `prod`.

```rust
use nestrs::prelude::*;

#[tokio::main]
async fn main() {
    NestFactory::create::<AppModule>()
        // Activates sanitization only in production environments
        .enable_production_errors_from_env()
        .listen(3000)
        .await;
}
```

## Production security checklist

<Note>
Before deploying a browser-facing or public API, verify each of these:

- **CORS**: replace `CorsOptions::permissive()` with an explicit origin allowlist
- **Security headers**: call `use_security_headers(SecurityHeaders::helmet_like())` and set a CSP and HSTS value
- **Rate limiting**: configure `use_rate_limit` (and optionally enforce limits at the edge)
- **Body size**: call `use_body_limit(max_bytes)` on public endpoints
- **Timeouts**: call `use_request_timeout(Duration::from_secs(n))` on public endpoints
- **CSRF**: if you use cookie auth in a browser flow, enable the `csrf` feature and wire `use_cookies()` + `use_csrf_protection`
- **Production errors**: call `enable_production_errors_from_env()` so stack traces never reach clients
- **TLS**: terminate TLS at a reverse proxy or at the process; nestrs does not handle TLS directly
- **Secrets**: load secrets from environment variables or a secret manager — never commit them
- **Dependency audit**: run `cargo audit` before releases
</Note>
