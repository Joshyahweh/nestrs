---
title: "Validate requests with DTOs, pipes, and extractors"
description: "Use nestrs's #[dto] macro and validator attributes to validate request bodies, query strings, and path params. Invalid input returns 422 automatically."
---

nestrs gives you compile-time and runtime validation in one step. You define a struct with the `#[dto]` macro, annotate fields with NestJS-style validator attributes like `#[IsEmail]` or `#[Length]`, and then use `ValidatedBody<T>`, `ValidatedQuery<T>`, or `ValidatedPath<T>` as handler extractors. When a request arrives with invalid data, nestrs automatically rejects it with a `422 Unprocessable Entity` response that includes a structured `constraints` map — no extra wiring required.

## The `#[dto]` macro

The `#[dto]` attribute macro transforms a plain struct into a fully wired DTO. It derives `serde::Deserialize`, `serde::Serialize`, `validator::Validate`, and `NestDto` in one shot. By default it also applies `#[serde(deny_unknown_fields)]`, so any JSON key that is not declared on the struct will fail deserialization immediately.

```rust
use nestrs::prelude::*;

#[dto]
pub struct CreateUserDto {
    #[IsString]
    #[Length(min = 1, max = 100)]
    pub name: String,

    #[IsEmail]
    pub email: String,

    #[Min(18)]
    pub age: u32,
}
```

<Note>
Extra JSON keys are **rejected by default** — `#[dto]` adds `#[serde(deny_unknown_fields)]` automatically. If you intentionally need to accept unknown keys (for example, a pass-through proxy body), use `#[dto(allow_unknown_fields)]`.
</Note>

## Validator attributes

The following attributes translate to `validator` crate constraints at compile time:

| Attribute | What it validates | Rust `validator` mapping |
|---|---|---|
| `#[IsString]` | Type-level marker (no-op; Rust types enforce this) | — |
| `#[IsEmail]` | RFC 5322 email address | `#[validate(email)]` |
| `#[IsUrl]` | URL format | `#[validate(url)]` |
| `#[IsUUID]` | UUID format | `#[validate(uuid)]` |
| `#[IsNotEmpty]` | Non-empty string | `#[validate(length(min = 1))]` |
| `#[Length(min = n, max = m)]` | String length between `n` and `m` | `#[validate(length(min = n, max = m))]` |
| `#[MinLength(n)]` | String length ≥ `n` | `#[validate(length(min = n))]` |
| `#[MaxLength(m)]` | String length ≤ `m` | `#[validate(length(max = m))]` |
| `#[Min(n)]` | Numeric value ≥ `n` | `#[validate(range(min = n))]` |
| `#[Max(m)]` | Numeric value ≤ `m` | `#[validate(range(max = m))]` |
| `#[IsPositive]` | Numeric value ≥ 1 | `#[validate(range(min = 1))]` |
| `#[IsNegative]` | Numeric value ≤ -1 | `#[validate(range(max = -1))]` |
| `#[ValidateNested]` | Recurse into a nested DTO | `#[validate(nested)]` |

## Validated extractors

Use these Axum extractors in your handler parameters instead of the bare `axum::Json<T>` or `axum::extract::Query<T>`. They deserialize the input **and** run the `validator::Validate` implementation generated by `#[dto]` before your handler body runs.

<Tabs>
  <Tab title="ValidatedBody">
    Deserializes a JSON request body and validates it. On failure, responds with `422` before the handler is called.

    ```rust
    use nestrs::prelude::*;

    #[dto]
    pub struct CreateUserDto {
        #[IsString]
        #[Length(min = 1, max = 100)]
        pub name: String,

        #[IsEmail]
        pub email: String,

        #[Min(18)]
        pub age: u32,
    }

    #[post("/users")]
    async fn create_user(
        ValidatedBody(dto): ValidatedBody<CreateUserDto>,
    ) -> Json<serde_json::Value> {
        Json(serde_json::json!({ "name": dto.name, "email": dto.email }))
    }
    ```
  </Tab>
  <Tab title="ValidatedQuery">
    Parses and validates query string parameters.

    ```rust
    use nestrs::prelude::*;

    #[dto]
    pub struct SearchQuery {
        #[Length(min = 1, max = 200)]
        pub q: String,

        #[Min(1)]
        pub page: u32,
    }

    #[get("/search")]
    async fn search(
        ValidatedQuery(query): ValidatedQuery<SearchQuery>,
    ) -> Json<serde_json::Value> {
        Json(serde_json::json!({ "q": query.q, "page": query.page }))
    }
    ```
  </Tab>
  <Tab title="ValidatedPath">
    Extracts and validates path parameters.

    ```rust
    use nestrs::prelude::*;

    #[dto]
    pub struct UserParams {
        #[Min(1)]
        pub id: u64,
    }

    #[get("/users/:id")]
    async fn get_user(
        ValidatedPath(params): ValidatedPath<UserParams>,
    ) -> Json<serde_json::Value> {
        Json(serde_json::json!({ "id": params.id }))
    }
    ```
  </Tab>
</Tabs>

## ValidationPipe

`ValidationPipe` implements `PipeTransform<T>` for any `T: validator::Validate`. You can apply it explicitly via `#[use_pipes(ValidationPipe)]` on a handler or controller when you need validation wired through the pipe pipeline rather than through an extractor.

```rust
use nestrs::prelude::*;

#[post("/items")]
#[use_pipes(ValidationPipe)]
async fn create_item(
    // dto is validated through the pipe before the handler runs
    ValidatedBody(dto): ValidatedBody<CreateItemDto>,
) -> Json<serde_json::Value> {
    Json(serde_json::json!({ "ok": true }))
}
```

## ParseIntPipe

`ParseIntPipe` transforms a `String` (for example from a path segment captured as a string) into an `i64`. It returns `400 Bad Request` if parsing fails.

```rust
use nestrs::prelude::*;

#[get("/orders/:id")]
async fn get_order(
    // Axum captures :id as a String; ParseIntPipe converts it
    #[use_pipes(ParseIntPipe)] id: i64,
) -> Json<serde_json::Value> {
    Json(serde_json::json!({ "id": id }))
}
```

## Nested DTOs

Use `#[ValidateNested]` together with a nested `#[dto]` struct to recursively validate complex bodies.

```rust
use nestrs::prelude::*;

#[dto]
pub struct AddressDto {
    #[Length(min = 1, max = 200)]
    pub street: String,

    #[Length(min = 2, max = 2)]
    pub country_code: String,
}

#[dto]
pub struct OrderDto {
    #[IsEmail]
    pub email: String,

    #[ValidateNested]
    pub address: AddressDto,
}
```

## Validation error shape

When a `ValidatedBody`, `ValidatedQuery`, or `ValidatedPath` extractor fails, nestrs returns `422 Unprocessable Entity` with a JSON body structured as:

```json
{
  "statusCode": 422,
  "message": "Validation failed",
  "error": "Unprocessable Entity",
  "details": [
    {
      "field": "email",
      "constraints": {
        "email": "email"
      }
    }
  ]
}
```

<Tip>
The `details` array maps each invalid field to a `constraints` object keyed by constraint name. You can use this shape directly to render field-level error messages in a form UI.
</Tip>
