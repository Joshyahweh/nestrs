---
title: "nestrs quickstart: build your first Rust HTTP API"
description: "Scaffold a nestrs project, write a module and controller, run the server, and add metrics, health checks, and request tracing in minutes."
---

This guide walks you from an empty directory to a running nestrs HTTP API. You will create a project, write a minimal module/controller/provider, start the server, and then layer on production features like metrics, health checks, and request tracing — all with real code from the framework itself.

## Prerequisites

You need the Rust toolchain installed (`rustup`, stable channel, Rust 1.88 or later). No other tools are required for the minimal path.

<Steps>

<Step title="Create a project">

You can scaffold a new project with the nestrs CLI or use plain Cargo. The CLI adds a production-ready `main.rs`, a `Dockerfile`, and a `.env.example` out of the box.

<Tabs>
  <Tab title="nestrs CLI (recommended)">
    ```bash
    cargo install nestrs-scaffold
    nestrs new my-api
    cd my-api
    ```

    The CLI creates the project directory, initializes a git repository, and writes a starter `src/main.rs` with a full `NestFactory` builder chain. See the [CLI reference](/cli) for all `nestrs new` options.
  </Tab>
  <Tab title="Cargo only">
    ```bash
    cargo new my-api
    cd my-api
    ```

    Then add dependencies manually in the next step.
  </Tab>
</Tabs>

</Step>

<Step title="Add dependencies">

Open `my-api/Cargo.toml` and add nestrs and its required companions. You need `tokio` for the async runtime and `serde` for serialization when you add DTO types later.

```toml
[dependencies]
nestrs = "0.3.8"
tokio = { version = "1", features = ["macros", "rt-multi-thread"] }
serde = { version = "1", features = ["derive"] }
```

<Note>
If you used `nestrs new`, your `Cargo.toml` is already populated — skip this step.
</Note>

</Step>

<Step title="Write the minimal application">

Replace the contents of `src/main.rs` with the following. This is the smallest useful nestrs application: one state provider, one controller, one route, and a `NestFactory` bootstrap.

```rust
use nestrs::prelude::*;

#[derive(Default)]
#[injectable]
struct AppState;

#[controller(prefix = "/api")]
struct HelloController;

#[routes(state = AppState)]
impl HelloController {
    #[get("/hello")]
    async fn hello() -> &'static str {
        "Hello from nestrs"
    }
}

#[module(controllers = [HelloController], providers = [AppState])]
struct AppModule;

#[tokio::main]
async fn main() {
    NestFactory::create::<AppModule>()
        .listen_graceful(3000)
        .await;
}
```

Here is what each part does:

- `#[injectable]` marks `AppState` as a dependency-injection provider.
- `#[controller(prefix = "/api")]` sets the route prefix for all routes in `HelloController`.
- `#[routes(state = AppState)]` wires the controller's `impl` block to the DI context.
- `#[get("/hello")]` registers an HTTP GET handler at `/api/hello`.
- `#[module(...)]` declares the application module, listing which controllers and providers it owns.
- `NestFactory::create::<AppModule>()` boots the application and `listen_graceful(3000)` starts the server with graceful shutdown on port 3000.

</Step>

<Step title="Run the server">

```bash
cargo run
```

Cargo compiles the project and starts the server. The first build takes longer while dependencies compile; subsequent builds are fast.

</Step>

<Step title="Test the endpoint">

Open your browser or use `curl`:

```bash
curl http://127.0.0.1:3000/api/hello
```

You should receive:

```
Hello from nestrs
```

<Tip>
The route path is composed from the controller prefix and the route attribute: `#[controller(prefix = "/api")]` + `#[get("/hello")]` → `GET /api/hello`. If you add `.set_global_prefix("v1")` to the factory chain, the full path becomes `/v1/api/hello`.
</Tip>

</Step>

<Step title="Add production enhancements">

The `NestFactory` builder chain lets you add cross-cutting features before calling `listen_graceful`. The code below is what `nestrs new` generates in `src/main.rs` for a production-ready starter:

```rust
#[tokio::main]
async fn main() {
    let port = std::env::var("PORT")
        .ok()
        .and_then(|v| v.parse::<u16>().ok())
        .unwrap_or(3000);

    NestFactory::create::<AppModule>()
        .set_global_prefix("api")
        .use_request_id()
        .use_request_tracing(RequestTracingOptions::builder().skip_paths(["/metrics"]))
        .enable_metrics("/metrics")
        .enable_health_check("/health")
        .enable_production_errors_from_env()
        .listen_graceful(port)
        .await;
}
```

| Builder method | What it adds |
|----------------|--------------|
| `.set_global_prefix("api")` | Prepends `/api` to every route |
| `.use_request_id()` | Attaches a unique `x-request-id` header to every request |
| `.use_request_tracing(...)` | Structured request/response logging; skip noisy paths like `/metrics` |
| `.enable_metrics("/metrics")` | Exposes a Prometheus-compatible `/metrics` endpoint |
| `.enable_health_check("/health")` | Exposes a `/health` liveness endpoint |
| `.enable_production_errors_from_env()` | Sanitizes 5xx response bodies in production (`NESTRS_ENV=production`) |

<Warning>
`.enable_metrics("/metrics")` exposes process and HTTP metrics publicly. In production, put this endpoint behind a network-level firewall or use `.use_request_tracing(RequestTracingOptions::builder().skip_paths(["/metrics"]))` to exclude it from access logs.
</Warning>

</Step>

</Steps>
