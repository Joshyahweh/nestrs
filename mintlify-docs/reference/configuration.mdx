nestrs is configured through three complementary mechanisms: Cargo feature flags that opt in to optional subsystems at compile time, environment variables that control runtime behaviour, and the `ConfigModule` / `load_config` pair for typed application configuration. This page covers all three.

---

## Cargo feature flags

Add features to your `Cargo.toml` dependency declaration. Only the features you enable are compiled into your binary.

```toml
[dependencies]
nestrs = { version = "0.x", features = ["openapi", "microservices", "microservices-nats"] }
```

### Transports and messaging

| Feature | Enables |
|---------|---------|
| `microservices` | TCP microservice transport, inter-process `EventBus`, and `NestFactory::create_microservice` |
| `microservices-nats` | NATS transport adapter (`NatsMicroserviceServer`, `NatsBrokerHealth`) |
| `microservices-redis` | Redis pub/sub transport adapter (`RedisMicroserviceServer`, `RedisBrokerHealth`) |
| `microservices-kafka` | Apache Kafka transport adapter (`KafkaMicroserviceServer`) |
| `microservices-mqtt` | MQTT transport adapter (`MqttMicroserviceServer`) |
| `microservices-rabbitmq` | RabbitMQ transport adapter (`RabbitMqMicroserviceServer`) |
| `microservices-grpc` | gRPC transport adapter (`GrpcMicroserviceServer`) — uses JSON-over-protobuf wire format |

<Note>The `microservices-*` transport features each imply `microservices`. You do not need to list both.</Note>

### Real-time and API surface

| Feature | Enables |
|---------|---------|
| `ws` | WebSocket gateway support via `nestrs-ws` (`#[ws_gateway]`, `#[ws_routes]`) |
| `graphql` | GraphQL endpoint via `nestrs-graphql` and `async-graphql` (`enable_graphql`) |
| `openapi` | OpenAPI JSON spec + Swagger UI via `nestrs-openapi` (`enable_openapi`, `#[openapi]`, `#[serialize]`) |

### Data and storage

| Feature | Enables |
|---------|---------|
| `cache-redis` | Redis-backed `CacheService` and distributed rate limiting in `RateLimitOptions::redis` |
| `database-sqlx` | `SqlxDatabaseModule` and `SqlxDatabaseService` (Postgres + SQLite via SQLx) |
| `mongo` | `MongoModule` and `MongoService` (MongoDB via the official Rust driver) |
| `http-client` | `HttpModule` and `HttpService` (reqwest-based HTTP client module) |

### Background work

| Feature | Enables |
|---------|---------|
| `schedule` | `ScheduleModule`, `ScheduleRuntime`, `#[cron]`, and `#[interval]` scheduled tasks |
| `queues` | `QueuesModule`, `QueuesService`, `QueuesRuntime`, and in-process job queues with `#[queue_processor]` |
| `queues-redis` | Redis-backed persistent queues (extends `queues`) |

### Cookie, session, and security

| Feature | Enables |
|---------|---------|
| `cookies` | `CookieManagerLayer` via `use_cookies()` |
| `csrf` | Double-submit CSRF middleware via `use_csrf_protection(config)` (implies `cookies`) |
| `session` | In-memory server-side sessions via `use_session_memory()` (implies `cookies`) |

### Rendering and files

| Feature | Enables |
|---------|---------|
| `mvc` | MVC template rendering via `MvcModule` and `MvcService` (minijinja templates) |
| `files` | Static file streaming helpers: `stream_file_octet_stream`, `stream_file_or_response`, `stream_file_with_content_type` |

### Observability

| Feature | Enables |
|---------|---------|
| `otel` | OpenTelemetry span export via OTLP (`try_init_tracing_opentelemetry`, `OpenTelemetryConfig`, `OtlpProtocol`) |

### Development and testing

| Feature | Enables |
|---------|---------|
| `test-hooks` | `RouteRegistry::clear_for_tests()` and `MetadataRegistry::clear_for_tests()` for resetting process-global state between integration tests |

<Warning>Never enable `test-hooks` in production binaries. Clearing the route and metadata registries at runtime breaks OpenAPI and metadata-driven guards. CI enables it automatically when running `cargo test --all-features`.</Warning>

## Environment variables

### Runtime environment detection

nestrs uses these variables to determine whether the process is running in production mode. The first non-empty variable wins, checked in order:

| Variable | Priority | Description |
|----------|----------|-------------|
| `NESTRS_ENV` | 1 (highest) | nestrs-specific env name. `production` or `prod` → production mode |
| `APP_ENV` | 2 | Generic application env variable |
| `RUST_ENV` | 3 (lowest) | Rust convention env variable |

Values are compared case-insensitively after trimming. Any value other than `production` or `prod` (for example `development`, `staging`, `test`) is treated as non-production. If none of the variables are set, `runtime_is_production()` returns `false` — safe for local development.

Production mode affects:
- `enable_production_errors_from_env()` — enables 5xx response sanitisation
- `enable_cors(CorsOptions::permissive())` — emits a `tracing::warn!` in production
- `load_config::<T>()` — skips `.env` file loading in production (reads only real environment variables)

### Logging and tracing

| Variable | Description |
|----------|-------------|
| `NESTRS_LOG` | Tracing filter directive. Overrides `RUST_LOG` when set and non-empty. Example: `"info,nestrs=debug,tower_http=warn"` |
| `RUST_LOG` | Standard Rust `tracing_subscriber` filter. Used when `NESTRS_LOG` is unset |

When neither variable is set, the tracing subscriber falls back to the `default_directive` you pass to `TracingConfig::builder().default_directive(...)` (which itself defaults to `"info"`).

```rust
let tracing = TracingConfig::builder()
    .format(TracingFormat::Json)
    .default_directive("info,nestrs=debug");

NestFactory::create::<AppModule>()
    .configure_tracing(tracing)
    .listen(3000)
    .await;
```

### Conventional variables

These are not read automatically by nestrs itself, but the `nestrs new` scaffold generates a `main.rs` that reads `PORT`:

| Variable | Conventional use |
|----------|-----------------|
| `PORT` | HTTP listen port (typically read in `main.rs` with `std::env::var("PORT").unwrap_or("3000")`) |

## ConfigModule and typed configuration

`ConfigModule` is a nestrs module that registers a single typed config struct as a singleton provider. Import it like any other module and inject your config type via `Arc<AppConfig>` in downstream providers.

### Step 1: define the config struct

Derive `serde::Deserialize`, `validator::Validate`, and `NestConfig`. Field names map directly to environment variable names (case-insensitive; `envy` handles underscore/uppercase normalisation automatically).

```rust
use nestrs::prelude::*;

#[derive(serde::Deserialize, validator::Validate, NestConfig)]
struct AppConfig {
    /// Required — no default. Fails at startup if DATABASE_URL is unset.
    database_url: String,

    #[validate(range(min = 1024, max = 65535))]
    port: u16,

    /// Optional with a default value via serde.
    #[serde(default = "default_log_level")]
    log_level: String,
}

fn default_log_level() -> String {
    "info".to_string()
}
```

### Step 2: import ConfigModule

```rust
#[module(
    imports = [ConfigModule::<AppConfig>],
    providers = [DatabaseService],
    controllers = [AppController],
)]
struct AppModule;
```

### Step 3: inject the config

```rust
#[injectable]
struct DatabaseService {
    config: Arc<AppConfig>,
}
```

`registry.get::<AppConfig>()` resolves the singleton loaded at startup. Field values come from environment variables (and `.env` files in non-production).

### load_config standalone

You can also call `load_config::<T>()` directly in `main` before the DI graph is built, for example to read the port for `listen`:

```rust
#[tokio::main]
async fn main() {
    let config = load_config::<AppConfig>().expect("invalid config");

    NestFactory::create::<AppModule>()
        .bind_all_interfaces()
        .listen_graceful(config.port)
        .await;
}
```

`load_config` loads `.env` and `.env.<env>` (e.g. `.env.development`) in non-production environments via `dotenvy`, then deserialises the environment using `envy`, then runs `validator::Validate`. If any required variable is missing or any validation rule fails, it returns a `ConfigError` with a descriptive message.

## NestConfig trait

`NestConfig` is a marker trait generated by the `#[NestConfig]` derive macro (or `#[derive(NestConfig)]`). It signals to `ConfigModule` that the type is a typed configuration struct. The trait itself has no methods — the real work is done by `serde::Deserialize` and `validator::Validate`, which must also be derived.

```rust
// What #[derive(NestConfig)] expands to (simplified):
impl nestrs::NestConfig for AppConfig {}
```

You need this marker when using `ConfigModule::<T>` in your module imports. For standalone use of `load_config`, the `NestConfig` bound is not required — any type that is `DeserializeOwned + Validate` works.

## Tracing and OpenTelemetry configuration

### configure_tracing

Installs the global `tracing` subscriber once (idempotent). Call it once in `main` before `listen`.

```rust
NestFactory::create::<AppModule>()
    .configure_tracing(
        TracingConfig::builder()
            .format(TracingFormat::Json)        // or TracingFormat::Pretty (default)
            .default_directive("info,nestrs=debug"),
    )
    .listen(3000)
    .await;
```

### configure_tracing_opentelemetry (feature: otel)

Same as `configure_tracing`, but additionally exports spans to an OTLP collector.

```rust
let otel = OpenTelemetryConfig::new("my-service")
    .endpoint("http://localhost:4317")
    .sample_ratio(1.0);

NestFactory::create::<AppModule>()
    .configure_tracing_opentelemetry(TracingConfig::builder(), otel)
    .listen(3000)
    .await;
```

On graceful shutdown (`listen_graceful` or `listen_with_shutdown`), nestrs calls `shutdown_tracer_provider()` automatically to flush pending spans before the process exits.
